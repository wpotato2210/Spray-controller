# DEVELOPMENT_ROADMAP.md

## Spray Controller Deterministic Roadmap (Arduino Nano)

| Phase | Objective | Deterministic Tasks | Deliverables (Done = all complete) |
|---|---|---|---|
| P0 – Baseline Audit Freeze | Lock current behavior before expansion. | 1) Freeze current module contracts in `include/interfaces.h`. 2) Confirm `src/main.cpp` loop cadence against `config.yaml:timing.loop_interval_ms`. 3) Record current assumptions: single flow sensor, single pump PWM, 3 binary sections. | - Audit note in `DEV_NOTES.md` with current loop path + module ownership.<br>- Interface checklist matching source files.<br>- No interface signature drift from baseline. |
| P1 – Pin + Config Finalization | Make runtime constants/pins authoritative and validated. | 1) Finalize `pins.yaml` for Nano-safe map (interrupt-capable pins for wheel/flow). 2) Finalize `config.yaml` keys for control gains, limits, timing. 3) Add deterministic YAML validation (`validation.yaml`) in CI/manual check. 4) Define one canonical loop interval (e.g., 50 ms) and enforce in firmware constants. | - Final `pins.yaml` + `config.yaml` schema with no TODO fields.<br>- Validation pass output (no missing/extra keys).<br>- `CONFIGURATION.md` and `HARDWARE.md` updated to exact values. |
| P2 – Core Control Loop Integration | Complete run-ready SEE→THINK→DO behavior. | 1) Integrate run/hold gating so HOLD forces pump minimum and output-safe section state. 2) Integrate global flow controller with active boom width from section manager. 3) Add loop determinism guard (elapsed-time check + fixed update order). 4) Publish compact serial status frame each loop interval. | - `src/main.cpp` executes stable deterministic cycle.<br>- `src/flow_controller.cpp`, `src/pump_control.cpp`, `src/section_manager.cpp` behavior aligned to spec.<br>- Protocol output documented and reproducible. |
| P3 – Sensor Robustness | Stabilize wheel + flow measurements; add optional pressure read-only input path. | 1) Add pulse timeout/zero-speed handling for wheel sensor. 2) Add flow plausibility clamp and stale-read fallback. 3) Add optional pressure sensor abstraction (`readPressure()`), telemetry only (no control coupling). 4) Define sensor fault flags for protocol/debug output. | - `flow_sensor` and `wheel_sensor` produce bounded values under no-pulse and burst conditions.<br>- Optional pressure module compile-time guarded and disabled by default.<br>- Fault bitfield/status text added to telemetry definition. |
| P4 – Operator Interface | Deliver usable field UI with safety confirmations. | 1) Implement menu navigation states. 2) Add live preview fields (speed, flow, duty, active sections). 3) Add distance + area accumulators based on wheel data and active width. 4) Add reset confirmation workflow for counters/calibration. | - Operator menu state machine documented and tested.<br>- Live preview updates at fixed interval.<br>- Distance/area reset requires explicit confirm action. |
| P5 – Scalability Preparation | Prepare architecture for more sections/sensors without logic rewrite. | 1) Replace hard-coded section arrays with config-driven section descriptors. 2) Introduce compile-time max constants (`MAX_SECTIONS`, `MAX_SENSORS`). 3) Isolate hardware adapters from control logic for future multi-MCU bridge. 4) Add deterministic ID mapping for section/sensor telemetry. | - Section count change requires config update only (no control algorithm rewrite).<br>- Interfaces remain backward compatible for current 3-section build.<br>- `ARCHITECTURE.md` updated with scalable module boundaries. |
| P6 – Calibration Workflow | Make calibration repeatable with optional manual override. | 1) Define flow calibration sequence (pulses/L or factor). 2) Define wheel calibration sequence (distance pulses). 3) Store/reload calibration constants with defaults and checksum/validity marker. 4) Add operator override flag (use defaults vs calibrated values). | - `CALIBRATION.md` converted from placeholder to executable steps.<br>- Calibration constants persisted and sanity-checked at boot.<br>- Override path tested and documented. |
| P7 – Documentation Freeze | Freeze implementation docs for Codex build reproducibility. | 1) Align `ARCHITECTURE.md`, `PROTOCOLS.md`, `USAGE.md`, `TESTING.md` with implemented firmware. 2) Replace placeholder diagrams in `DIAGRAMS.md` with final SEE/THINK/DO + wiring + state flow. 3) Mark frozen interface version and date. | - Doc set internally consistent (pins, protocol fields, units, states).<br>- No placeholder markers remain in core docs.<br>- Frozen-doc tag recorded in `CHANGELOG.md`. |
| P8 – Release Readiness | Produce stable tagged firmware release. | 1) Run full bench validation matrix (run/hold, section toggles, sensor disconnect, min/max speed). 2) Run YAML validation and config compatibility checks. 3) Bump firmware version, tag release, update changelog entry with exact date. 4) Archive known limitations. | - Release candidate passes all mandatory tests.<br>- Git tag + changelog + config versions aligned.<br>- Reproducible build/flash instructions verified in `INSTALL.md`. |
| P9 – Optional Enhancements | Non-blocking extensions after stable release. | 1) Multi-MCU transport abstraction (UART/CAN bridge). 2) Advanced UI (preset profiles, guided calibration). 3) Remote logging output channel. 4) GPS-driven area/speed fallback logic. | - Optional feature branch specs with acceptance criteria.<br>- No regression to baseline single-Nano firmware path. |

## Phase Gates (Execution Order)

1. **Must complete in order:** P0 → P1 → P2 → P3 → P4 → P5 → P6 → P7 → P8.
2. **P9 starts only after P8 tag exists.**
3. **Gate rule:** each phase closes only when all listed deliverables are present in repo and referenced from `CHANGELOG.md`.
